# 智能家居控制面板 - 开发规范

## 1. 目录结构规范

### 1.1 总体目录结构

```
├── index.html          # 入口页面
├── index.css           # 全局样式
├── index.js            # 主逻辑
├── config/             # 配置文件
│   ├── config.js       # 应用配置
│   └── device_config.js # 设备配置
├── app/                # 核心应用逻辑
│   ├── ha-connection.js    # HA连接
│   ├── state-manager.js    # 状态管理
│   ├── websocket-manager.js # WebSocket
│   └── device-controller.js # 设备控制
├── cards/              # 卡片组件
├── pages/              # 页面
│   ├── mobile/         # 移动端页面
│   └── desktop/        # 桌面端页面
├── apps/               # 应用目录
│   ├── apps.json       # 应用配置
│   ├── notes/          # 笔记应用
│   ├── inventory/      # 库存应用
│   └── voron/          # 打印机应用
├── libs/               # 第三方库
├── backgrounds/        # 背景主题
└── docs/               # 文档
```

### 1.2 应用目录结构

每个应用应遵循以下目录结构：

```
apps/
  应用名称/
    index.html          # 应用入口
    assets/             # 资源文件
      images/           # 图片
      icons/            # 图标
    js/                 # JavaScript文件
    css/                # 样式文件
    config/             # 应用配置
    README.md           # 应用说明文档
```

## 2. 命名规范

### 2.1 文件命名

- **HTML文件**：使用小写字母，单词之间用连字符(-)分隔，如 `home-page.html`
- **CSS文件**：使用小写字母，单词之间用连字符(-)分隔，如 `home-page.css`
- **JavaScript文件**：使用小写字母，单词之间用连字符(-)分隔，如 `device-controller.js`
- **配置文件**：使用小写字母，单词之间用下划线(_)分隔，如 `device_config.js`
- **资源文件**：使用小写字母，单词之间用下划线(_)分隔，如 `logo_icon.svg`

### 2.2 变量命名

- **JavaScript变量**：使用驼峰命名法，如 `deviceState`
- **CSS类名**：使用小写字母，单词之间用连字符(-)分隔，如 `app-container`
- **HTML元素ID**：使用小写字母，单词之间用下划线(_)分隔，如 `device_list`
- **常量**：使用全大写字母，单词之间用下划线(_)分隔，如 `MAX_RETRY_COUNT`

### 2.3 函数命名

- **普通函数**：使用驼峰命名法，如 `loadDeviceData()`
- **构造函数**：使用首字母大写的驼峰命名法，如 `DeviceController`
- **回调函数**：使用动词开头，如 `onDeviceClick()`
- **私有函数**：使用下划线开头，如 `_privateMethod()`

## 3. 代码风格规范

### 3.1 HTML规范

- 使用语义化HTML标签
- 缩进使用4个空格
- 标签名小写
- 属性值使用双引号
- 自闭合标签必须使用闭合符号
- 避免使用内联样式
- 为重要元素添加ID和class

### 3.2 CSS规范

- 使用BEM命名规范（Block, Element, Modifier）
- 缩进使用4个空格
- 每个选择器单独占一行
- 每个属性单独占一行
- 使用小写字母和连字符
- 避免使用!important
- 合理使用CSS变量

### 3.3 JavaScript规范

- 使用ES6+语法
- 缩进使用4个空格
- 每行代码长度不超过80个字符
- 使用单引号
- 大括号使用K&R风格
- 变量声明使用let/const
- 函数使用箭头函数（适当情况下）
- 避免使用全局变量
- 合理使用模块化

## 4. 开发流程规范

### 4.1 应用开发流程

1. **创建应用目录**：在 `apps/` 下创建应用目录
2. **创建应用文件**：创建 `index.html` 作为应用入口
3. **配置应用**：在 `apps.json` 中添加应用配置
4. **开发应用**：实现应用功能
5. **测试应用**：确保应用正常运行
6. **文档编写**：编写应用README.md文件

### 4.2 代码提交规范

- 提交信息要清晰明了
- 提交信息格式：`[类型] 描述`
  - 类型：feat(新功能)、fix(修复)、docs(文档)、style(样式)、refactor(重构)、test(测试)、chore(构建)
  - 描述：简洁描述本次提交的内容

### 4.3 测试规范

- 确保应用在不同设备上正常运行
- 测试应用的各项功能
- 测试应用的边界情况
- 测试应用的性能和响应速度

## 5. 应用集成规范

### 5.1 应用配置

在 `apps.json` 中添加应用配置：

```json
{
  "id": "app1",
  "name": "应用名称",
  "icon": "图标",
  "path": "应用路径"
}
```

- **id**：应用唯一标识符
- **name**：应用显示名称
- **icon**：应用图标（可以是emoji或SVG路径）
- **path**：应用入口文件路径

### 5.2 应用通信

应用与主框架之间的通信使用postMessage：

- **从应用发送消息**：`window.parent.postMessage({ type: '消息类型', data: '数据' }, '*')`
- **从主框架发送消息**：`iframe.contentWindow.postMessage({ type: '消息类型', data: '数据' }, '*')`

### 5.3 应用生命周期

- **初始化**：应用加载时接收 `app-container-init` 消息
- **菜单点击**：应用接收 `menu-click` 消息
- **设置应用名称**：应用发送 `set-app-name` 消息

## 11. 最佳实践

- 遵循DRY原则（Don't Repeat Yourself）
- 遵循KISS原则（Keep It Simple, Stupid）
- 编写可维护的代码
- 编写可测试的代码
- 编写可扩展的代码

## 12. 常见问题处理

- **应用加载失败**：检查应用路径和文件是否存在
- **应用显示异常**：检查CSS和JavaScript是否正确
- **应用通信失败**：检查postMessage的使用是否正确
- **性能问题**：检查是否有不必要的DOM操作或网络请求

本规范旨在确保项目的一致性、可维护性和可扩展性。所有开发者应严格遵守本规范，以确保项目的质量和稳定性。

随着项目的发展，本规范将不断更新和完善。
